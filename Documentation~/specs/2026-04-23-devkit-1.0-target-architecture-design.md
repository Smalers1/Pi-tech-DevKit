## 1. Strategic context

### 1.1 The billion-dollar product vision

Pi tech is building HealthOn XR into the definitive next-generation medical training platform: **high-fidelity XR natively powered by clinical AI**. The spatial immersion and the AI are one unified product, not two adjacent tracks. By December 2026 we operate as a multi-tenant enterprise platform with active university deployments and recurring revenue.

The three-year trajectory:

| Year | Capability |
|---|---|
| **2026** | Multi-tenant platform. VICKY as platform assistant. VICKY observes lab sessions and acts narrowly on author-declared action surfaces (Target B). |
| **2027** | VICKY inside the AR/VR lab — voice in, voice out, observes student actions, provides real-time clinical correction in student's language, analyzes performance. |
| **2028** | **Prompt-to-simulation.** A professor describes a clinical scenario in plain language; VICKY assembles it from a library of Building Blocks (rooms, equipment, patients, scenarios, interactables, analytics). The professor reviews, tweaks, publishes. |

**Every line of code written today either makes 2028 easier or harder.** This document's architecture is designed so that 2027 capabilities are an additive delta, 2028 capabilities are a role + metadata-index expansion, and no year requires re-architecture.

### 1.2 Why DevKit is the leverage point

The HealthOn XR ecosystem has five surfaces: Web Portal, Mobile App, VR App (HealthOn VR), AR App (HealthOn AR), DevKit. All of them consume DevKit directly or indirectly. **DevKit is the substrate through which the 2028 Building Blocks library manifests.** Every reusable prefab we ship today, every Step type, every ConsoleAction, every validator becomes vocabulary for VICKY's future authoring capabilities.

This gives DevKit a dual mandate:

1. **Primary:** make Pi tech engineers build AR/VR medical training labs dramatically faster and more reliably. This is DevKit's daily reason to exist.
2. **Strategic:** be the architectural substrate VICKY inhabits as she moves from platform assistant (2026) to embedded AI instructor (2027) to lab author (2028).

The design in this document satisfies both, with primary winning when they conflict.

### 1.3 Why v1.0 now

DevKit is at v0.10.1. It has carried the product this far through a feature-split codebase (Core / ContentDelivery / Scenario / Interactables / Quiz / Stats / UI), two monolithic runtime classes (`SceneManager.cs` at 2,505 lines, `ContentDeliverySpawner.cs` at 1,180 lines), reflection-heavy extension points, and zero public-API discipline. It works. It does not scale to a six-month VICKY-in-AR/VR milestone or to the 2028 prompt-to-simulation vision without re-architecture.

The cost of doing this wrong is not a slow shipment — it is building the next two years of features on unstable foundations and paying compound interest on technical debt at exactly the moment the product needs to scale across universities. The cost of doing it right is 5–7 months of disciplined migration work with zero behavior break and zero consumer surprise.

## 2. Guiding principles

These are the non-negotiable properties of the v1.0 architecture. Every subsequent section of this document defends against them.

1. **Systems thinking, not module thinking.** Split by role in the system (Domain, Runtime, Capabilities, Agent, Bridge, Consumers, Authoring), not by feature area (Quiz, Stats, Selectables). A Quiz exists as a typed Domain object, a runtime runner, a capability binding, and an authoring surface — one thing per layer, not one bundle across layers.
2. **Addressable-portable by construction.** Labs are prefabs delivered by CCD, spawned into a host scene. Lab code cannot assume scene context, cannot call `Camera.main`, cannot `GameObject.Find`. Everything the lab needs comes from the capability registry the host provides.
3. **No hardcoding, ever.** Reflection-by-string, `#ifdef` branches in runtime code, index-based RPCs, magic scene names, hand-rolled JSON serialization, `Debug.Log("[Scenario]…")` prefixes — all banned. Replaced with typed registries, capability interfaces, stable ids, `[SerializeReference]` + `MovedFromAttribute`, and `IDevKitLogger` channels.
4. **Non-breaking evolution.** AR and VR ship to universities on v0.10 today. v0.11 → v1.0 introduces zero shipped-content breaks, guaranteed by `[MovedFrom]` + `FormerlySerializedAs` on every relocation (with a prefab-override validator catching Unity's known `[MovedFrom]` gap), a facade preserving the current public API surface, and seven CI gates (public API diff, v0.10 fixture load, golden event replay, zero-alloc, dependency direction, banned APIs, IL2CPP round-trip) run on every PR.
5. **VICKY-legible by construction.** Domain types are POCOs round-trippable to JSON. Every agent-actionable entity has a stable id. The agent substrate is a typed observer + gated action surface that reuses the same Lab Console dispatch path the professor panel already runs through. VICKY integration is a capability registration, not a new subsystem.
6. **Quality first, no rush.** Each phase ships when its CI gates are green and its exit criteria are met. A slipping phase pauses rather than skipping gates. Rollback is always one `manifest.json` line.
7. **Developer experience is a product.** DevKit's editor tooling — Hub, Project Setup Tool, Building Blocks, Scenario Simulator, Live State Inspector, Validators — is as much a product surface as the runtime. UI Toolkit with a shared design system. Every editor window inherits a base class with theme, help footer, version badge. New engineer Day-3 onboarding time is a metric we measure.
8. **Every choice serves multiple purposes.** If an architectural decision only serves VICKY, it's weighted wrong. A well-chosen boundary serves authoring productivity (purpose 1), runtime correctness (purpose 2), testability (purpose 8), and VICKY-legibility (purpose 9) simultaneously. Those are the choices we make.

**Enabler:** 8 (serves 1, 2, 7, and 10).

No layer of the architecture serves only one purpose. No purpose is served by only one layer.

## 5. Cardinal constraints (summary)

These are the facts of life the architecture is built under.

1. **v0.10 is live in production universities.** Breaking a shipped AR or VR lab mid-semester is a customer incident.
2. **Labs are Addressables-delivered prefabs spawned into a host scene.** No scene-embedded lab logic is allowed; all lab behavior comes from the prefab and its capabilities from the host.
3. **Quest 2 performance budget is the baseline.** 72 FPS sustained, ≤ 200 batches, ≤ 500K triangles, zero GC allocations per frame in hot paths. Reference: handbook §6.11.
4. **EU AI Act Annex III §3 + GDPR Article 22** apply to VICKY-in-lab. Explicit consent, transparency, revocation, audit export are non-negotiable at the agent substrate boundary.
5. **Fusion 2 in Shared Mode is the default multiplayer.** `[Networked]` state with `OnInput`, not RPCs for state. Reference: handbook §6.7.
6. **Every reusable prefab ships with a metadata sidecar.** `BuildingBlockMetadataV1` schema. This is the currency of the 2028 prompt-to-simulation system and a handbook Non-Negotiable.
7. **Naming convention is enforced.** `Category_Subtype_Variant_v1` for prefabs, `lab.[labId].[type]` for Addressables groups, `lab/[labId]/[type]/[name]` for addresses. Validators fail PRs that break convention.
8. **No service keys in clients.** RLS always. Anon key only. Every DevKit integration with Supabase respects this.
9. **Unity 6 LTS is the engine.** DevKit v1.0 targets Unity 6 (minimum `6000.0`; recommended `6000.3` LTS). ~~HealthOn AR is still on Unity 2022.3~~ **CORRECTED (Petros 2026-06-10, §28.6): HealthOn AR and HealthOn VR are BOTH already on Unity 6; Vuforia is fine — no consumer upgrade dependency.** The `package.json` `unity`-field bump (2022.3 → 6000.0) + the `dependencies` block land together at the **Phase D (post-launch) metadata cutover** (§28.6), NOT in the behaviour-neutral launch Phase A; consumer repos pin a specific git tag until they adopt.
10. **Unity 6 AOT/IL2CPP discipline.** Unity 6's Awaitable is zero-allocation on frame-aware waits and is the preferred async primitive for DevKit's public API (no external dependency). UniTask may be used internally where its wider API surface pays off, but consumer-facing interfaces return `Awaitable<T>` so AR/VR repos don't need to pull UniTask. IL2CPP remains the ship-to-device scripting backend; all AOT caveats in §7.5 apply.

**Dependency rule:** a layer may reference only lower-numbered layers. Authoring (7) references Domain (1) at the data layer; tooling may reference Capabilities interfaces for preview and simulation. No layer references Consumers (6). Domain (1) references nothing DevKit-specific.

**Enforced by:** `.asmdef` references + a CI gate that parses the asmdef graph and fails on any cycle or upward reference.

### 6.2 Why this split and not the existing feature split

The current (v0.10) module structure is `Core / ContentDelivery / Scenario / Interactables / Quiz / Stats / UI`. That is a feature split — it groups things the product team describes, not things the system distinguishes.

Concretely, `Quiz` in v0.10 is simultaneously: a Domain type (`QuizAsset`, `QuizStep`), a runtime behavior (`QuizSession` state machine, step runner embedded in `SceneManager.cs`), a UI binding (`QuizUIController`), and an editor tool (`QuizAssetEditor`). They ship bundled in one asmdef. That is why `SceneManager.cs` must import `QuizUIController`, why step runner extraction cannot proceed without editing a god-class, why testing one without loading the others is impossible.

The layered split gives `Quiz` exactly one concern per layer:

- **Domain:** `QuizAsset` POCO, `QuizStep`, `MiniQuizStep`, `QuizResultsStep`, `QuizQuestion`, `QuizSessionState` (pure data)
- **Runtime:** `QuizStepRunner`, `MiniQuizStepRunner`, `QuizResultsStepRunner` (one per file, one responsibility)
- **Capabilities:** `IQuizPanelService` interface; default UI Toolkit-based impl
- **Authoring:** `QuizAssetInspector`, `QuizAssetValidator` (editor-only)

Each has one job. Each is testable independently. Adding a new quiz variant means dropping a single Domain POCO + a single Runner + (optional) a single inspector. Nothing else changes. Compare to v0.10: adding a new step type today touches four files minimum and risks breaking the shared dispatch.

## 7. Layer 1 — Domain

Domain is the only layer whose contracts cross every trust boundary. Authoring writes it. Runtime executes it. Capabilities produce and consume it. Agent substrate observes and mutates it. Telemetry snapshots it. The web portal (eventually) authors it. Consumer repos serialize it. If Domain is wrong, every layer above pays.

### 7.1 Rules

- Zero MonoBehaviour. Zero coroutine. Zero `UnityEngine.UI`, `UnityEngine.Networking`, or any Unity runtime type beyond the small set of value types (`Vector3`, `Color`, `Rect`) that are legitimately shared and serialization attributes (`[SerializeReference]`, `[FormerlySerializedAs]`, `[MovedFrom]`).
- Every type that crosses a trust boundary has a `SchemaVersionV1` attribute and is named with a `V1` suffix (`LabObservationV1`, `LaunchContextV1`, etc.). Immutable after publish; `V2` ships alongside with explicit migration code.
- All polymorphic lists (`Scenario.steps`, `MiniQuizStep.questions`, etc.) use `[SerializeReference]` in Unity and a `type` discriminator in JSON.
- No Unity-scene references in types that need JSON round-trip. Scene-bound references (Transforms, Directors, Buttons) live on Domain types used only by the authoring layer; the runtime projects them to stable-id observation shapes for VICKY.

### 7.2 The authoring-vs-observation split

This is the key insight that makes VICKY integration tractable without breaking authoring ergonomics.

A `QuestionStep` in Unity editor has:

```csharp
[Serializable]
public class QuestionStep : Step
{
    public RectTransform panelRoot;        // scene-bound — for authoring
    public Animator panelAnimator;
    public List<Choice> choices;           // each Choice has a Button ref
    // ... other authoring fields
}
```

At runtime, `QuestionStepRunner.Project(step, state)` produces a stable-id observation:

```json
{
  "version": "v1",
  "type": "question",
  "guid": "step-abc",
  "panel": { "id": "panel.intro-question", "visible": true },
  "choices": [
    { "id": "choice.yes", "label": "Yes",  "enabled": true, "next": "step-def" },
    { "id": "choice.no",  "label": "No",   "enabled": true, "next": "step-ghi" }
  ]
}
```

VICKY sees the observation. It references `choice.yes` when it wants to highlight or suggest. It never sees the `Button` reference. Authors drag buttons into Choice slots exactly as they do in v0.10; the runtime derives a clean, actionable view for the agent.

### 7.3 Identity model — stable ids for agent-actionable entities

Every entity VICKY might observe or actuate needs an author-assigned stable id. Convention: `lowercase.dot.separated` with semantic prefixes (`panel.intro-question`, `choice.symptom.yes`, `param.vitals.heart-rate`).

| Entity | v0.10 has id? | v1.0 has id? |
|---|---|---|
| Step | ✓ `guid` | keep |
| Selectable | ✓ id field | keep |
| Panel (`panelRoot`) | ✗ (inferred from GameObject name) | **add `string id`** |
| Choice button (Question, MiniQuiz) | ✗ | **add `string id`** |
| Cue card | ✗ (index-based) | add `string id` (optional; index is acceptable fallback) |
| Stat key | ✓ string | keep |
| Quiz question | ✓ `questionId` | keep |
| Lab (Scenario) | ✗ | **add `string labId` linked to `LaunchContextV1.labId`** |
| ConsoleParameter | — | `string id` from creation |
| ConsoleAction | — | `string id` from creation |

Ids are author-assigned (default-generated in editor on first save with a "rename me" hint), validated by `IdUniquenessValidator` at save time (unique within scope), immutable across runtime sessions. Adding ids is backward-compatible: v0.10 prefabs without ids receive defaults on first editor open; shipped runtime data is not touched.

### 7.5 Serialization

- **In-memory authoring:** `[SerializeReference]` for polymorphic step lists; `[SerializeField]` with `[FormerlySerializedAs]` for field renames.
- **JSON for cross-boundary contracts:** `com.unity.nuget.newtonsoft-json` (Unity's official UPM distribution) with **hand-authored polymorphic converters keyed on a `"type"` discriminator** (not Newtonsoft's built-in `$type` mechanism, which has AOT/IL2CPP edge cases). Shape: `{ "version": "v1", "type": "…", ... }`.
- **IL2CPP AOT safety (the known landmine):** Newtonsoft.Json on IL2CPP/ARM64 has long-standing pitfalls around generics and the `$type` polymorphism mechanism — derived types can deserialize as base, and `[Preserve]` does not cover all generic specializations. Mitigation shipped in DevKit v1.0:
  1. Hand-authored converter per polymorphic root (no `$type` reliance); converters live in `Runtime/Domain/Serialization/Converters/`.
  2. A `link.xml` fragment under `Runtime/Domain/` preserving every V1 contract type (`<type fullname="Pitech.XR.Domain.…" preserve="all" />`).
  3. `[UnityEngine.Scripting.Preserve]` attribute on each schema class.
  4. `AotHelper.EnsureType<T>()` calls at `RuntimeInitializeOnLoadMethod(SubsystemRegistration)` time for every generic specialization used by the converters (`List<Choice>`, `Dictionary<string, ParameterValue>`, etc.).
  5. **CI Gate 7 (new) — IL2CPP round-trip test:** a PlayMode test in `Tests/PlayMode/Serialization.Tests/` round-trips every V1 contract through an IL2CPP-built player target (not only the editor). Added to the CI gate stack in §15.3. This is the test that would have caught Android-build-only failures in the current codebase.
- **Schema version enforcement:** `SchemaVersion` attribute on the class + runtime check on deserialization. Mismatched version fails loudly with guidance to run the migrator.

### 7.6 Consequences for the rest of the architecture

Because Domain is pure data with typed contracts:

- Every other layer tests against Domain without Unity runtime
- VICKY's mental model of a lab is exactly the JSON it sees — no hidden Unity state
- Web portal (eventually) authors the same types by producing conformant JSON
- Migration is provable (schema-version check catches stale content immediately)
- The 2028 prompt-to-simulation VICKY reads `BuildingBlockMetadataV1` + `ScenarioV1` + `LabConsoleV1` from a tenant-scoped library; it doesn't need to understand Unity

### 7.7 Objectives and Constraints

Authors declare what counts as success. The runtime measures it. The portal reports it. Authors never write evaluation logic; they pick from a polymorphic criterion family and the runtime + (where required) VICKY evaluate against the same author-declared definition.

**Vocabulary:** the engineer-facing concept is **Objective** (`Domain/Objectives/`). The portal's buyer-facing presentation can call this "Competence" or "Learning Outcome" — both vocabularies map to the same underlying `ObjectiveOutcomeV1` envelope. The DevKit type names are deliberately game-dev-native ("define the objectives of a lab"); the portal handles domain framing for accreditation audiences.

**Three tiers, one pipeline.** Different scenarios need different evaluation strategies, but all three flow through the same `LabEventBus` → `TelemetryService` → portal pipeline. The discriminator is **who knows the success criteria** — the author at design time, or VICKY at runtime.

| Tier | Name | Who declares | Who evaluates | Typical use | Reliability |
|---|---|---|---|---|---|
| **1** | `Tier1_Deterministic` | Author (full criteria) | Runtime evaluator | Linear procedural scenarios; placement, timing, sequence, quiz, action correctness | High; no AI |
| **2** | `Tier2_StateGoal` | Author (goal states + constraints) | Runtime evaluator | Dynamic scenarios with known outcome states (emergency response, multi-path procedures) | High; no AI |
| **3** | `Tier3_AiEvaluated` | Author (evaluation prompt + category) | VICKY in Examiner mode | Judgment-laden criteria: clinical reasoning, communication, situational awareness | Lower; requires professor confirmation per EU AI Act Art. 22 (see §10.6 + §10.9) |

Roughly 80% of competence signal comes from Tier 1, 15% from Tier 2, 5% from Tier 3 — but Tier 3 is the highest-value, highest-defensibility layer because no competitor without an embedded agent can produce it.

**Domain subtree:**

```
Domain/Objectives/
  LabObjectiveV1.cs              ← top-level: what counts as overall lab success
  StepObjectiveV1.cs             ← per-step success conditions
  ObjectiveCriterionV1.cs        ← base type, polymorphic
  ObjectiveTier.cs               ← Tier1_Deterministic | Tier2_StateGoal | Tier3_AiEvaluated
  ObjectiveCategory.cs           ← Procedural | Knowledge | Judgment | Communication | Safety
  ObjectiveWeight.cs             ← aggregation weight (for overall lab scoring; weights sum to 1.0)

  Criteria/                      ← Tier 1 (deterministic, author-declared)
    PlacementAccuracyCriterion.cs    ← distance from target ≤ tolerance
    TimeBoundCriterion.cs            ← completed within / under N seconds
    SequenceCorrectnessCriterion.cs  ← steps performed in prescribed order
    QuizAccuracyCriterion.cs         ← answer matches expected
    ActionCorrectnessCriterion.cs    ← action invoked with correct args
    ObjectStateCriterion.cs          ← e.g., "IV bag connected at lab end"

  StateGoals/                    ← Tier 2 (state-based, dynamic scenarios)
    GoalStateCriterion.cs            ← "patient stabilized"
    RequiredActionSetCriterion.cs    ← "must perform CPR + call code + defibrillate, in any order"
    TimeWindowCriterion.cs           ← "within 5 minutes of trigger event"

  Constraints/                   ← anti-objectives (must NOT happen)
    NoTouchConstraint.cs             ← sterile field violation
    NoMissedSafetyCheckConstraint.cs
    NoContraindicatedActionConstraint.cs

  Evaluation/
    ObjectiveEvaluationProposalV1.cs ← Tier 3 — VICKY's proposal with evidence citations
    ObjectiveOutcomeV1.cs            ← finalized per-objective result (met/failed/partial + score)
    AttemptSummaryV1.cs              ← roll-up of all objectives for an attempt
```

**Authoring contract.** Author attaches `StepObjectiveV1` instances to `Step` (Scenario layer), `ConsoleObjective` to `LabConsole` (Console layer), or `LabObjectiveV1` to the lab itself. Each carries one or more `ObjectiveCriterionV1` and an `ObjectiveWeight`. Constraints are a separate polymorphic family (anti-objectives — violation publishes `ConstraintViolatedEvent`, see §8.1).

**Evaluation contract.** Tiers 1 and 2 are evaluated deterministically by the `IObjectiveEvaluator` capability (see §9.2 / §9.5 / §9.9). Tier 3 routes to the Agent Substrate via the existing observation pipeline; Examiner mode emits `ObjectiveEvaluationProposalV1` (see §10.9). Professor confirmation in the portal finalizes a `ObjectiveOutcomeV1`.

**No new transport.** All objective outcomes ride the existing `LabEventBus` → `TelemetryService` → bridge → portal pipeline. The objective layer is an evaluator + a typed event family; it does not introduce a parallel data plane.

**Authoring guardrails.** Three validators (`ObjectiveWeightSumValidator`, `Tier3PromptNonEmptyValidator`, `ObjectiveCoverageValidator`) ship in the editor — see §13. They prevent shipping labs with malformed objective definitions (weights that don't sum to 1.0, empty Tier 3 prompts, linear scenarios with un-evaluated steps).

## 8. Layer 2 — Runtime

Runtime is where Domain data becomes behavior. It contains the step-type-agnostic orchestrator, the Lab Console parameter store and action dispatcher, the step runners, the effect handlers, the event bus, the registries, and the bootstrap pattern.

### 8.1 The LabEventBus — the central nervous system

Every runtime interaction flows through one typed, in-process, tenant-scoped pub/sub:

```csharp
public interface ILabEvent { LabId LabId { get; } DateTime AtUtc { get; } }

public readonly struct StepEnteredEvent          : ILabEvent { /* guid, kind */ }
public readonly struct StepProgressedEvent       : ILabEvent { /* guid, kind, progress01 */ }
public readonly struct StepExitedEvent           : ILabEvent { /* guid, kind, outcomeNextGuid */ }
public readonly struct ParameterChangedEvent     : ILabEvent { /* paramId, oldValue, newValue, actor, role, revision */ }
public readonly struct ActionInvokedEvent        : ILabEvent { /* actionId, args, actor, role, outcome, latencyMs */ }
public readonly struct SelectionChangedEvent     : ILabEvent { /* listId, selected, correct, wrong */ }
public readonly struct QuizAnsweredEvent         : ILabEvent { /* questionId, choiceId, correct */ }
public readonly struct ObservationRequestedEvent : ILabEvent { /* since, observer */ }
public readonly struct ConsentGateEvent          : ILabEvent { /* outcome, actionId, reason */ }

// Objective / learning-semantic events (see §7.7). Published by IObjectiveEvaluator
// for Tier 1 + Tier 2 deterministically; published by Agent Substrate (Examiner mode)
// for Tier 3 proposals. All ride the same telemetry pipeline as the lifecycle events
// above — no separate transport.
public readonly struct ObjectiveMetEvent              : ILabEvent { /* objectiveId, tier, score, evidence */ }
public readonly struct ObjectiveFailedEvent           : ILabEvent { /* objectiveId, tier, reason, evidence */ }
public readonly struct ConstraintViolatedEvent        : ILabEvent { /* constraintId, severity, evidence */ }
public readonly struct HintRequestedEvent             : ILabEvent { /* stepGuid, hintLevel, actor */ }
public readonly struct AttemptCompletedEvent          : ILabEvent { /* attemptId, summary (AttemptSummaryV1), durationMs, outcome */ }
public readonly struct ObjectiveEvaluationProposedEvent : ILabEvent { /* objectiveId, proposal (ObjectiveEvaluationProposalV1), vickyConfidence, evidence, citations */ }

public interface ILabEventBus
{
    IDisposable Subscribe<TEvent>(Action<TEvent> handler) where TEvent : ILabEvent;
    void Publish<TEvent>(in TEvent evt) where TEvent : ILabEvent;
}
```

**Zero-alloc contract:** events are `readonly struct`, subscription tables are pre-allocated per event type, `Publish` dispatches without closure capture, `Subscribe` returns an `IDisposable` backed by a struct. An EditMode test asserts zero bytes allocated in a publish loop.

**One bus per `LabRoot`:** when a lab despawns, the bus is disposed, all subscriptions drop, no cross-lab event leakage is possible.

**Eight types of subscriber:** step runners (to coordinate), Lab Console (to apply effects), telemetry capability, replication capability, VICKY observer, professor panel UI, debuggers (Live State Inspector, Action Log Viewer), and the objective evaluator (subscribes to lifecycle/parameter/action events to publish `ObjectiveMetEvent` / `ObjectiveFailedEvent` / `ConstraintViolatedEvent`). Tests subscribe too.

### 8.2 The registries — the single extension mechanism

Six registries, every one attribute-driven, discovered at domain-reload once, cached statically.

| Registry | Attribute | Add a feature by |
|---|---|---|
| `StepRunnerRegistry` | `[StepRunner(typeof(MyStep))]` | Drop `MyStep` in Domain/Steps + `MyStepRunner : IStepRunner<MyStep>` in ScenarioRuntime. |
| `EffectHandlerRegistry` | `[EffectHandler(typeof(MyEffect))]` | Drop `MyEffect` in Domain/Console + `MyEffectHandler : IEffectHandler<MyEffect>` in LabConsoleRuntime. |
| `ValidatorRegistry` | `[DevKitValidator(order = N)]` | Drop a class implementing `IDevKitValidator` in Authoring.Core. |
| `BuildingBlockRegistry` | `[BuildingBlock("name", category = "...")]` | Drop a template prefab + a descriptor class in Authoring.BuildingBlocks. |
| `CapabilityRegistry` | `[Capability(typeof(IFoo), priority = N, scope = App|Lab)]` | Drop an impl of the interface with the attribute anywhere in runtime assemblies; or register explicitly via `DevKitBootstrapper`. |
| `InspectorControlRegistry` | `[InspectorControl(typeof(MyParameter))]` | Drop a UI Toolkit visual element for a custom parameter type. |

**No switch statements anywhere in core code.** The `ScenarioRunner.Run` loop has one branch — "what's the runner for this step type?" — and answers it via registry lookup, not type-testing.

**IL2CPP safety:** every discovered type is referenced via `[Preserve]` and a generated `link.xml` fragment so the IL2CPP stripper doesn't eliminate it.

### 8.3 The bootstrap pattern

Two components, two scopes.

**Per-app (consumer-owned):** `DevKitBootstrapper` subclass in the host scene. Registers concrete capability impls for that consumer. Runs in `Awake()` with `[DefaultExecutionOrder(-1000)]`.

**Per-lab (DevKit-owned):** `LabRoot` component on every lab prefab root. When the prefab spawns:

1. Create lab-scoped `LabEventBus`
2. Compose lab-scope capabilities over app-scope registry
3. Discover children: `Scenario` (linear), `LabConsole` (dynamic), others
4. Instantiate `ScenarioRunner` if `Scenario` present; `LabConsoleRuntime` if `LabConsole` present
5. Connect telemetry subscriber, replication subscriber, agent substrate (if `LaunchContext.vickyMode != Absent`)
6. Start runner and runtime in declared order

When the prefab despawns: dispose `LabEventBus`, unsubscribe everything, release capabilities with lab scope.

**`XRServices` compatibility shim:** calls to `XRServices.Register<T>(x)` and `XRServices.Get<T>()` delegate to the capability registry. Existing v0.10 consumer code keeps working unchanged (Rule 3 of §13).

### 8.4 ScenarioRunner — step-type-agnostic orchestrator

```csharp
public sealed class ScenarioRunner
{
    public async Awaitable Run(Scenario scenario, ScenarioRunContext ctx, CancellationToken ct)
    {
        int idx = 0;
        while (idx >= 0 && idx < scenario.steps.Count && !ct.IsCancellationRequested)
        {
            var step = scenario.steps[idx];
            if (step == null) { idx++; continue; }

            var runner = StepRunnerRegistry.Get(step.GetType());
            ctx.Bus.Publish(new StepEnteredEvent(step.guid, step.Kind, ctx.LabId, DateTime.UtcNow));

            var outcome = await runner.Run(step, ctx, ct);
            ctx.Bus.Publish(new StepExitedEvent(step.guid, step.Kind, outcome.nextGuid, ctx.LabId, DateTime.UtcNow));

            idx = ResolveNext(outcome.nextGuid, idx, scenario);
        }
    }
}
```

150 lines of orchestration. No step-type knowledge. No hidden couplings. Testable with fake step runners and a fake bus.

### 8.5 IStepRunner — the step contract

```csharp
public interface IStepRunner<TStep> where TStep : Step
{
    Awaitable<StepOutcome> Run(TStep step, StepContext ctx, CancellationToken ct);
    StepObservation Project(TStep step, StepRunState state);
}

public readonly struct StepContext
{
    public ILabEventBus Bus { get; }
    public CapabilityRegistry Capabilities { get; }
    public LabId LabId { get; }
    public LabConsoleRuntime? Console { get; }
}

public readonly struct StepOutcome
{
    public string nextGuid;
    public StepResult result; // Completed, Cancelled, Skipped, Errored
}
```

Every runner:

- Does its work through `ctx.Capabilities` — no `Camera.main`, no `GameObject.Find`
- Emits progress events on `ctx.Bus`
- Projects an observation on demand for VICKY
- Cancellable via `ct` and fully awaits the cancellation before returning

**Async primitive choice — `Awaitable<T>` over `UniTask<T>`.** Unity 6's built-in `Awaitable` / `Awaitable<T>` is a zero-allocation, frame-aware async type with the same shape as UniTask for 90% of our needs (`NextFrameAsync`, `WaitForSecondsAsync`, `FromCancellationToken`). Using it in DevKit's public API means:

- **Zero external dependency for consumers** (AR and VR don't need to pull UniTask through DevKit)
- **Zero-alloc frame-aware waits** match the handbook §6.11 budget by default
- **First-class Unity lifetime integration** — cancellation on scene unload, domain reload, play-mode exit is native

Internally, runners may use UniTask where its broader API (`WhenAll`, `WhenAny`, `Forget`, `Delay` with cancel-on-destroy) pays off in authoring ergonomics. The public signature stays `Awaitable<T>`; the internal body can `await task.AsUniTask()` or stay on Awaitable as the author prefers. A `UniTaskAwaitableAdapter` lives in `Runtime/Core/Async/` for the adapter case.

**No reflection anywhere in the dispatch path.** All effect handlers are typed against their `EffectDescriptor`. All action descriptors are registered with typed arg schemas.

### 8.7 The GroupExecutor — lifted out of the monolith

Group steps run multiple nested steps concurrently with N-of-M completion rules. In v0.10 this logic is embedded in `SceneManager.cs` (lines ~1583–2305). In v1.0 it extracts as a standalone `GroupExecutor` with:

- `GroupCancelToken` for coordinated cancellation
- `GroupChildHandle` tracking completion state per child
- Registration of required/optional children per mode
- Multi-condition branch evaluation

`GroupStepRunner` uses `GroupExecutor` to implement its `Run` method. Tests cover every `CompleteWhen` mode and every cancellation path.

### 8.8 ConditionsStep — reflection eliminated

v0.10 uses runtime reflection (`GetFieldOrPropertyValue`, `ReadStringMember`, `ReadNumericMember`) to read arbitrary MonoBehaviour fields/properties by string name. This is fragile (typos become silent zero), slow, and IL2CPP-risky.

v1.0 replaces with typed `IConditionValueSource` implementations:

- `StatValueSource` — reads from `IStatsService`
- `ComponentFieldValueSource<TComponent, TValue>` — generated or authored typed reader
- `LabeledListValueSource<TItem>` — typed list reader with key lookup

Authors bind a typed source at design time via a custom inspector that generates the appropriate adapter. Conditions evaluate with zero reflection at runtime.

### 8.9 Performance budgets

| Component | Budget |
|---|---|
| `LabEventBus.Publish<T>` | 0 alloc, < 1µs per call |
| `ScenarioRunner.Run` main loop | 0 alloc per iteration |
| `StepRunner.Project` | 0 alloc (writes into pre-allocated `StepObservation` buffer) |

## 9. Layer 3 — Capabilities

### 9.1 The core principle

A lab is an Addressables-delivered prefab spawned into a host scene. The lab doesn't own the camera, input rig, canvas, audio mixer, network peer, or clock — the host does. **The lab asks for them through capability interfaces.** The host registers concrete implementations once per app via `DevKitBootstrapper`. Labs access them through `LabRoot.Capabilities`.

This is what makes purpose #3 (cross-platform capability abstraction) structurally true. It is also what allows purposes #2 (runtime correctness) and #7 (consumer stability) to hold across ARFoundation migrations, VR middleware swaps, and future Web XR hosts.

### 9.3 Host-context resolution pattern

Consumers (AR, VR, Simulator) subclass `DevKitBootstrapper` and register concrete capability impls:

- **HealthOn AR (Vuforia today):** `ARVuforiaCameraService`, `ScreenSpaceCanvasPanelHost`, `VuforiaGroundPlaneAnchor`, `TouchAndPointerInputService`, `TouchRaySelection`, `LocalReplicationService`
- **HealthOn AR (ARFoundation future):** swap `ARVuforiaCameraService` → `ARFoundationCameraService`, `VuforiaGroundPlaneAnchor` → `ARFoundationPlaneAnchor`. Everything else identical. Zero lab code changes.
- **HealthOn VR:** `XRRigCameraService` (OVR), `WorldSpaceCanvasPanelHost`, `XRControllerInputService`, `XRRaySelection`, `MetaHapticsService`, `FusionReplicationService`

### 9.4 Graceful degradation

- **No-op defaults** for non-critical capabilities (`IHapticsService`, `ILocalizationService`) mean labs never null-check.
- **TryGet pattern** for truly optional capabilities (`ILabObserver`, `ILabActionSurface`): `if (caps.TryGet<ILabObserver>(out var obs)) obs.Emit(snapshot);`. Only three capabilities use this pattern.

### 9.5 Scopes

- **App-scope:** registered once in `DevKitBootstrapper.ConfigureCapabilities`. Lifetime = app. Examples: `ICameraService`, `IAudioService`, `ITelemetryService`, `INetworkReplicationService`, `IObjectiveEvaluator` (single instance subscribes to per-lab `LabEventBus` via a small adapter at `LabRoot` spawn time).
- **Lab-scope:** created by `LabRoot` at spawn, disposed at despawn. Lifetime = one lab. Examples: `ILabObserver` (per-lab VICKY bridge), `IAuditTrail` (per-lab records), `LabConsoleRuntime` itself.

`LabRoot.Capabilities` is a composite: first lab-scope, then app-scope fallback. Labs see one registry.

### 9.6 The banned-API Roslyn analyzer

A `Pitech.XR.Analyzers` package runs in CI and forbids scene-context APIs in lab runtime code:

- `Camera.main`
- `GameObject.Find`, `FindObjectsOfType<T>`, `FindAnyObjectByType<T>`
- `UnityEngine.SceneManagement.SceneManager.GetActiveScene()`
- `EventSystem.current` lookup
- `Canvas.FindObjectsOfType` / tag-based lookup for Canvas
- `PhotonNetwork.*`, `FusionNetworkRunner.*` direct reference from runtime (allowed only in `Capabilities.VR.Fusion`)
- Reflection against consumer types (`GetType().GetField(name)` in runtime code)

Scanned assemblies: `ScenarioRuntime`, `LabConsoleRuntime`, `Capabilities.Interfaces`, any class tagged `[InLabScope]`.

**Deployment:** P1 ships as warnings; P2 promotes to hard CI gate after any P1 violations are fixed.

### 9.7 Content delivery as a capability

The v0.10 `ContentDeliverySpawner` (1,180 lines) decomposes across layers:

- `IContentDeliveryService` (interface) — Capabilities/Interfaces
- `AddressablesContentDeliveryService` (default impl) — Capabilities/Defaults
- `ContentDeliveryFlowStateMachine` (pure state machine, testable without Unity) — Domain/Content
- `DefaultContentDeliveryOverlayController` (UI binding) — Capabilities/Defaults (uses `IPanelHostService`)
- `ContentDeliverySpawnerEditor` (authoring UI) — Editor/ContentDelivery.Editor (existing)

The in-scene MonoBehaviour `ContentDeliverySpawner` keeps its v0.10 public surface (facade) and internally delegates to this decomposition.

**CCD cohort pinning via badges** — the concrete mechanism for purpose #4 (cohort-scale content delivery). Unity CCD's **badge** system is the production-ready primitive: each release in a CCD bucket can carry a named badge (e.g., `aigaiou-fall-2026`, `production`, `staging`); the client requests content by badge name via `CcdManager.Environment / Bucket / Badge` at launch. Our `LaunchContextV1` already carries a `resolvedVersionId` field; the web portal's `resolve-launch-metadata` edge function maps `(cohort, lab)` to the correct badge and returns it in the launch payload. AR/VR clients read it and configure `CcdManager` before `Addressables.LoadContentCatalog`. Cohort pinning = the portal leaving a cohort's badge pinned to a specific release while other cohorts auto-advance. Mid-semester surprise updates are prevented by construction.

### 9.8 Fusion 2 replication — StateAuthority semantics (Shared Mode)

**Context.** VR labs run Fusion 2 in Shared Mode (handbook §6.7). In Shared Mode, each `NetworkObject` has exactly one **StateAuthority** at a time. Networked Properties synchronize **from StateAuthority to all other clients**. A non-authority client writing to a `[Networked]` property produces only local prediction, which the StateAuthority can override in the next tick. **This has direct consequences for how `LabConsoleRuntime` and `GatedLabActionSurface` must be wired in multiplayer.**

**The correct dispatch pattern:**

```
Peer with local UI/VICKY writes to a ConsoleParameter
            │
            ▼
LabConsoleRuntime.Set<T>(...) on the local peer
            │
            ▼
        Is local peer the StateAuthority for this LabConsole's NetworkObject?
            │
        ┌───┴──────────────────────────────────────────┐
       YES                                             NO
        │                                              │
        ▼                                              ▼
   Write [Networked] properties                  INetworkReplicationService.
   directly; gates already passed;               SendToAuthority(actionId, args,
   replication happens automatically             origin-actor, origin-role)
   via Fusion.                                   as an RPC.
        │                                              │
        │                                              ▼
        │                                   Authority receives RPC;
        │                                   **re-runs** the 5-gate pipeline
        │                                   against the relayed actor/role
        │                                   (trust-but-verify — never trust
        │                                   a relayed gate pass);
        │                                   on accept, writes the [Networked]
        │                                   properties, which replicate back.
        │                                              │
        └──────────────────────┬───────────────────────┘
                               ▼
                    All peers observe the change via Fusion ChangeDetector;
                    each peer's local LabEventBus publishes
                    ParameterChangedEvent; subscribers (telemetry,
                    observer projection, professor panel UI) react.
```

**Consequences for `INetworkReplicationService`:**

```csharp
public interface INetworkReplicationService
{
    bool IsLocalAuthority(NetworkObjectId id);
    Awaitable SendToAuthority(ActionRequest request, CancellationToken ct);
    IDisposable OnRemoteChange<T>(Action<T> handler);
}
```

Default (`LocalReplicationService`): `IsLocalAuthority` always true; `SendToAuthority` is a local call. No-op for single-player AR/VR.

Fusion (`FusionReplicationService` in the optional `Capabilities.VR.Fusion` sub-package): wraps a Fusion `NetworkBehaviour` whose `[Networked]` properties mirror the `LabConsole`'s parameter state; `SendToAuthority` dispatches a Fusion RPC tagged `RpcSources.All, RpcTargets.StateAuthority`; `OnRemoteChange` hooks into Fusion's `ChangeDetector`. Getters/setters for `[Networked]` props are empty (Fusion code-generates them). Tick rate 30 Hz per handbook §6.7. **Pin:** Photon Fusion `2.0.12` (stable, March 2026) or the latest stable 2.0.x. **Not 2.1.x preview** — those carry known blocker bugs and API drift during the preview phase. Fusion 2 is a Unity Verified Solution with Unity 6 compatibility confirmed.

**Trust-but-verify on the authority.** An important invariant: when the authority receives an `ActionRequest` RPC from a non-authority peer, it **re-runs the 5 gates** (Consent, Role, Rate, ContentSafety, AuditAvailability) against the **relayed** actor and role, never trusts that the origin gated correctly. This blocks a malicious or bugged peer from claiming a higher VickyMode than the launch context grants it. Audit records the origin actor AND the authority's re-verification outcome.

**Why not a simpler `AllBuffered` RPC (v0.10's ControlOptionManager pattern)?** Because `AllBuffered` treats every client as equally authoritative, which (a) fails trust-but-verify (any client can force a state change), (b) has no single source of truth for conflict resolution (two clients writing the same param at the same tick produce last-writer-wins with undefined ordering), and (c) inflates bandwidth for every state change rather than letting Fusion's `ChangeDetector` economize. The Shared Mode StateAuthority pattern is the correct Fusion 2 idiom.

**Authority selection.** For VR labs: the host peer (`NetworkRunner.IsSharedModeMasterClient`) owns StateAuthority for the LabConsole NetworkObject by default. This works for typical classroom topologies (one professor is host, students are peers). Authority transfer is possible via Fusion's `NetworkObject.RequestStateAuthority` but is not exercised in v1.0.

### 9.9 Objective evaluator capability

`IObjectiveEvaluator` is the runtime arm of the Objectives & Constraints model defined in §7.7. App-scope, single instance per app, subscribes to per-lab `LabEventBus` via a small adapter at `LabRoot` spawn. No host-specific overrides — the same `DefaultObjectiveEvaluator` runs in AR, VR, and the Simulator host.

**Tier 1 (deterministic):** subscribes to `StepExitedEvent`, `ParameterChangedEvent`, `ActionInvokedEvent`, `QuizAnsweredEvent`, `SelectionChangedEvent`. For each event, looks up the matching `StepObjectiveV1` / `ConsoleObjective`, evaluates each `ObjectiveCriterionV1` against current state, and publishes `ObjectiveMetEvent` or `ObjectiveFailedEvent`.

**Tier 2 (state goals):** maintains a per-lab state-tracker that watches goal-state predicates (`GoalStateCriterion`, `RequiredActionSetCriterion`, `TimeWindowCriterion`). Publishes `ObjectiveMetEvent` on goal transition; publishes `ConstraintViolatedEvent` on constraint breach.

**Tier 3 (AI-evaluated):** does NOT evaluate locally. Instead, registers a hook with the Agent Substrate (when present) to request observation projection at the trigger boundary the author declared (e.g., end of diagnosis phase, or on a specific `StepExitedEvent`). VICKY in Examiner mode consumes the snapshot and emits `ObjectiveEvaluationProposedEvent` (see §10.9). When the agent substrate is absent (Simulator host without VICKY, or `vickyMode == Absent`), Tier 3 objectives are recorded as `pending_human_review` and surface in the portal queue.

**End-of-attempt aggregation:** on `StepExitedEvent` matching the scenario's terminal step (or on explicit `ScenarioRunner.Complete()`), the evaluator builds an `AttemptSummaryV1` rolling up all `ObjectiveOutcomeV1` records and publishes `AttemptCompletedEvent`. Telemetry capability picks this up like any other event.

**Zero-alloc on the hot path:** the evaluator pre-allocates per-lab tracker buffers at spawn; per-event evaluation writes into pre-allocated outcome structs; no closure capture in subscribe callbacks. Same allocation budget as the existing telemetry subscriber.

### 9.10 Telemetry capability — two production implementations

`ITelemetryService` exposes one interface. Three implementations ship; consumers pick one:

| Impl | Where it runs | Transport | Lives in |
|---|---|---|---|
| `BridgeQueuedTelemetryService` | AR consumer (mobile UaaL host) | Unity → mobile UaaL bridge → mobile's `telemetry-queue.service.ts` → cloud | `Capabilities.Defaults` |
| `DirectCloudQueuedTelemetryService` | VR consumer (Quest standalone) | Quest standalone → embedded queue → Supabase HTTPS | `Capabilities.VR.Telemetry` (sub-package, see §11.5) |
| `InMemoryTelemetryService` | Simulator host | In-memory ring buffer (no network) | `Capabilities.Defaults` |

All three:
- Consume the **same** `TelemetryEventV1` envelope (no per-impl schema variants)
- Share an **offline-queue core** (`Capabilities.Defaults.Telemetry.Core/`) with identical retry strategy (exponential backoff with jitter), batching window, flush triggers (size threshold, time threshold, app foreground/background), and auth model (anon key + session token)
- Emit identical `TelemetryEventV1` payloads end-to-end so portal ingestion is one path; analytics queries do not branch by surface

The mobile `telemetry-queue.service.ts` behavior is the canonical reference. `DirectCloudQueuedTelemetryService` is a port — same retry semantics, same idempotency keys, same offline-buffer eviction policy. Future surfaces (web shell, alternate VR headsets) implement the same interface against the same core; analytics-side code never changes.

## 10. Layer 4 — Agent Substrate

### 10.1 The single insight

VICKY is not a special system. **VICKY is another client of the Lab Console with three additions: a consent gate, a role filter, and an audit trail.** Everything else — observation, action dispatch, telemetry, replication — uses the same code paths the professor panel and multiplayer peers already run.

This is why the 6-month VICKY-in-AR/VR milestone lands as a delta over the substrate, not a new parallel subsystem. It is also why the 2027 voice-in/voice-out + in-scene presence and the 2028 prompt-to-simulation are additive expansions, not re-architectures.

### 10.2 VickyMode

```csharp
public enum VickyMode
{
    Absent,     // VICKY is not present in this lab session
    Tutor,      // VICKY guides, hints, explains; does not evaluate
    Examiner,   // VICKY evaluates, corrects after the fact; does not pre-empt
    Helper,     // VICKY answers direct questions from the learner
    Observer    // VICKY watches silently; used for review, analytics, transparency demos
}
```

- `VickyMode` is **orthogonal to platform `user_role`** (student / professor / admin). `user_role` governs who is allowed to pick a `VickyMode` for a session. `VickyMode` governs what VICKY does during a session.
- The learner selects `VickyMode` at lab launch via mobile app UI. The choice goes into `LaunchContextV1.vickyMode`.
- Authors declare which actions are available in which modes via `[AllowedModes(VickyMode.Tutor, VickyMode.Helper)]` on `ConsoleAction` subclasses.
- Future modes can be added by extending the enum (additive, non-breaking).

### 10.3 The six components and the gate pipeline

```
        ILabObserver               ILabActionSurface
        (read path)                (write path, gated)
             │                              │
             ▼                              ▼
      ObservationProjector         GatedLabActionSurface
      builds LabObservationV1      runs the gate pipeline:
      from the event bus             1. ConsentGate
             │                        2. RoleFilter (VickyMode)
             ▼                        3. RateLimiter
      UaaLBridgeObserver              4. ContentSafetyGate (no-op in v1.0)
      emits to mobile via              5. AuditTrail.Write (pending)
      the bridge                       6. LabConsoleRuntime.Invoke
                                       7. AuditTrail.Update (outcome)
                                       8. TelemetryService.Emit
```

**Gate outcomes:** each gate returns accept / reject with a typed reason code. Rejection short-circuits the pipeline; audit still records the attempt with the rejection reason. Accepted invocations reach dispatch; `AuditTrail` updates the outcome after dispatch returns.

**Fail-closed:** any gate unavailable → reject. There is no "skip the gate" path.

**Typed errors:** `CONSENT_REQUIRED`, `ROLE_FORBIDDEN`, `RATE_LIMITED`, `CONTENT_SAFETY_BLOCKED`, `AUDIT_UNAVAILABLE` — enum values mobile and portal handle explicitly.

### 10.4 The role-filtered tool catalog — generated, not hand-written

Authors never write VICKY tool definitions. They declare `ConsoleAction` classes; the catalog falls out.

```csharp
[ConsoleAction(id = "offer_hint_current_step")]
[AllowedModes(VickyMode.Tutor, VickyMode.Helper)]
[ArgSchema(typeof(HintArgsV1))]
[Description("Highlights the currently-correct Selectable and plays a subtle audio cue. Use when the learner has been idle on the current step for >15s.")]
public sealed class OfferHintCurrentStep : ConsoleAction<HintArgsV1>
{
    public override void Execute(HintArgsV1 args, ActionContext ctx)
    {
        ctx.Capabilities.Get<ISelectablesRuntime>().Highlight(args.SelectableId, 2f, args.Intensity);
        ctx.Capabilities.Get<IAudioService>().Play("cues/hint_subtle");
    }
}
```

At lab load, `AgentToolCatalogBuilder`:

1. Reflects over registered `ConsoleAction` types
2. Filters by `LaunchContextV1.vickyMode` vs each action's `[AllowedModes]`
3. Emits `AgentToolCatalogV1` with id, description, arg schema, mode tag per action
4. Pushes to mobile via `AgentToolInvocationBridge`

Mobile VICKY's edge function injects this into VICKY's tool list. VICKY's LLM emits tool calls against it. Tool call returns over the bridge → `GatedLabActionSurface.Invoke` → gate pipeline → dispatch.

**You write zero VICKY-specific glue.** One `ConsoleAction` declaration exposes to: professor panel UI, VICKY tool catalog, audit trail, telemetry, and (when C-mode unlocks) prompt-to-simulation composition.

### 10.5 Observation — pull, don't push, zero-alloc

`ObservationProjector` does not run per frame. It subscribes to semantically-significant events on the bus (`StepEnteredEvent`, `StepExitedEvent`, `ParameterChangedEvent` above a configured significance threshold). On its own schedule (default 500 ms, configurable per lab), it composes a `LabObservationV1` snapshot by reading current state from each step runner's `Project` method and from `LabConsoleRuntime.Observe`. The snapshot writes into a reusable pre-allocated buffer.

Only then is the snapshot pushed to the `UaaLBridgeObserver`, which marshals it to mobile. Mobile VICKY sees a fresh snapshot every 500 ms plus event-driven extras when something meaningful changes.

**Frame budget for observation:** < 0.5 ms/frame on Quest 2. Verified in PlayMode tests with Oculus Performance HUD integration.

### 10.6 Consent gate — EU AI Act + GDPR compliance made concrete

**Regulatory frame (governing, not advisory):**

- **EU AI Act (Regulation (EU) 2024/1689)** — VICKY-in-lab is classified as **high-risk AI under Annex III §3** (AI in education and vocational training, including AI used to evaluate learning outcomes or steer student progress). High-risk obligations become enforceable on **2027-12-02** (post Digital Omnibus trilogue 2026-05-07 — deferred from earlier 2026-08-02). Article 13 (transparency obligations to deployers/users) and Article 27 (Fundamental Rights Impact Assessment, "FRIA") apply. Engineering maintains counsel-ready posture for 1.0 launch procurement (target 2026-09-01 per `VICKY_1_0_LAUNCH.md`); full Conformity Assessment is post-1.0 work under HXR-262.
- **GDPR (Regulation (EU) 2016/679)** — any AI system processing personal data triggers GDPR. Article 13 (information to data subject), Article 15 (right of access), Article 22 (automated decision-making), Article 35 (Data Protection Impact Assessment, "DPIA") all apply. The GDPR DPIA and AI Act FRIA are complementary, not alternatives — both are required.
- **Penalty framing:** AI Act fines up to 3% of global revenue for high-risk non-compliance; GDPR fines up to 4%. Multiplicative — compounds matter.

**What `DefaultConsentGate` delivers in v1.0:**

1. **Explicit per-session opt-in** (EU AI Act Art. 13 + GDPR Art. 13) when the learner picks `VickyMode` at lab launch. The `ConsentRecordV1` records timestamp, learner identifier (pseudonymized per GDPR), chosen `VickyMode`, list of actions that mode enables with human-readable descriptions, and hash of the transparency notice text version shown.
2. **Transparency notice** (AI Act Art. 13) surfaced in the mobile app before the learner confirms. Lists actions VICKY may take, data that flows where, and the fact that VICKY is an AI system (AI Act Art. 50 — "informed when necessary"). Notice text is versioned; policy changes produce a new version hash and re-prompt on next session.
3. **Revocation** (GDPR Art. 7(3) — right to withdraw consent) via a visible "stop VICKY" control during the lab. Revoking disables `ILabActionSurface` within the current tick (not next frame — the tick). Emits `ConsentRevokedEvent`; surface resumes only after explicit re-opt-in. Pre-revocation work (observations, audit records) is retained; post-revocation actions are impossible by construction.
4. **Right of access** (GDPR Art. 15) via a learner-facing audit export endpoint in the portal, returning the full `ConsoleAuditSink` record for the learner's sessions in a machine-readable format.
5. **Automated decision-making safeguards** (GDPR Art. 22) — no `VickyMode.Examiner` action may produce a *binding* assessment outcome without professor review. Examiner mode **proposes** evaluations; professor **confirms**. Proposals are stored separately from confirmed evaluations in the portal; the confirmation step is an explicit human-in-the-loop checkpoint.

**What lives outside the consent gate (portal + organizational):**

- DPIA document (GDPR Art. 35) and FRIA document (AI Act Art. 27) — authored per-tenant, stored in the portal's compliance area, not in DevKit
- Technical documentation (AI Act Art. 11 + Annex IV) — this document (spec + per-phase plans + operational runbooks) satisfies the technical-documentation obligation
- Post-market monitoring (AI Act Art. 72) — implemented via the `tool_call_events` + `grounding_decision` telemetry pipeline in the portal

All four runtime items live in `DefaultConsentGate` in v1.0. Mobile app surfaces (1)–(3) at lab launch and mid-session. Audit export (4) extends the portal's existing `tool_call_events` pipeline with a per-learner export endpoint. Automated-decision safeguard (5) is a `VickyMode.Examiner`-specific policy in `RoleFilter`.

**Enforcement date pressure.** **2027-12-02** is the regulatory deadline for Annex III high-risk systems (post Digital Omnibus trilogue 2026-05-07 deferral; earlier 2026-08-02 framing DEFUNCT). For 1.0 launch (2026-09-10), the binding requirement is **counsel-ready posture for university procurement** (target 2026-09-01 per `VICKY_1_0_LAUNCH.md` §3.4): Phase 1 artifacts shipped 2026-05-21 (PIT-346/PIT-363/PIT-485..491 — see ROADMAP §0.5); counsel sign-off on Art 13 + Art 14 templates + Greek Art 57 sandbox entry needed by Sep 1. Full Conformity Assessment + FRIA/DPIA delivery = post-1.0 work under HXR-262 Post-launch backlog (2026-10-15 → 2026-12-15). For DevKit specifically: consent flow, technical-documentation per Art 11, and post-market monitoring per Art 72 are 1.0 in-scope; Conformity-Assessment-grade artifacts are Phase 2.

### 10.7 Rate limiting and circuit breaking

- Per-action rate limits (e.g., `offer_hint_current_step` at most once per 30s per lab)
- Per-lab global limits (e.g., ≤ 30 actions/minute — catches VICKY reasoning loops)
- Circuit breaker: consent revoked mid-lab disables `ILabActionSurface` within the tick; emits `ConsentRevokedEvent`; surface resumes only after explicit re-opt-in

Every year's capability is a role-policy + action-declaration + metadata-index expansion. The substrate is data-driven; adding new authoring primitives means adding declarations, not rewriting dispatch.

### 10.9 Tier 3 objective evaluation flow (Examiner mode)

`VickyMode.Examiner` is the substrate for Tier 3 (AI-evaluated) objectives defined in §7.7. The flow is additive over the existing observation + gated-action pipeline — no new substrate, no new transport.

**Trigger.** The author declares a Tier 3 objective with a trigger boundary (e.g., "end of diagnosis phase" referenced by a step `guid`, or "on `StepExitedEvent` for step X"). At runtime, `IObjectiveEvaluator` (§9.9) detects the trigger and requests an observation snapshot via the existing `ObservationProjector` (§10.5).

**Evaluation.** The agent substrate consumes the `LabObservationV1` snapshot, evaluates against the author's Tier 3 prompt (e.g., "evaluate clinical reasoning during the diagnosis phase"), and produces an `ObjectiveEvaluationProposalV1` with:
- evaluation verdict (met / failed / partial)
- numeric score within author-defined range
- evidence citations referencing specific observation field paths and event timestamps
- VICKY's confidence (0.0–1.0)
- the original objective `id` and `prompt` (for portal display)

**Emission.** Proposal is published as `ObjectiveEvaluationProposedEvent` on the `LabEventBus`. Telemetry capability picks it up like any other event; the bridge marshals it to the portal alongside the rest of the telemetry stream.

**Confirmation gate (GDPR Art. 22, EU AI Act Art. 14).** Per §10.6 item 5, no Examiner-mode evaluation is binding without professor confirmation. The portal stores the proposal in a `pending_human_review` queue separately from confirmed `ObjectiveOutcomeV1` records. Professor confirms or rejects in the portal UI; on confirmation, the proposal becomes a finalized `ObjectiveOutcomeV1` with a `confirmedBy` field. Rejected proposals are retained for audit but do not contribute to competence reporting.

**Failure modes:**
- **VICKY unavailable / `vickyMode == Absent`:** Tier 3 objectives are recorded as `pending_human_review` with no proposal payload — surfaces in the portal queue for direct professor evaluation.
- **VICKY low-confidence (< author-defined floor):** proposal is still emitted but flagged `low_confidence`; portal can surface these prominently in the review queue.
- **Consent revoked mid-lab before trigger:** Tier 3 objective is skipped, recorded as `consent_revoked_pre_trigger`; deterministic Tier 1/2 objectives unaffected.
- **Observation projection fails:** evaluator records `observation_unavailable`; objective remains `pending_human_review`.

**No autonomous grading.** VICKY never finalizes an `ObjectiveOutcomeV1` directly. The Article 22 safeguard is structural — the portal schema does not allow Examiner-emitted proposals to be written to the finalized outcome table without a `confirmedBy` user reference. This is the architectural enforcement of "VICKY proposes, professor confirms."

### 11.5 VR telemetry transport (Quest standalone)

§11.1–§11.4 describe the AR / Mobile UaaL bridge. VR consumers (Quest standalone) have no mobile UaaL host and need a direct cloud telemetry path. The interface and envelope stay identical to AR — only the transport differs.

**Architecture:**

| Aspect | AR (Mobile UaaL) | VR (Quest standalone) |
|---|---|---|
| Interface | `ITelemetryService` | `ITelemetryService` *(same)* |
| Envelope | `TelemetryEventV1` | `TelemetryEventV1` *(same)* |
| Implementation | `BridgeQueuedTelemetryService` | `DirectCloudQueuedTelemetryService` |
| Transport | Unity → mobile UaaL bridge → mobile's `telemetry-queue.service.ts` → Supabase | Quest standalone → embedded queue → Supabase HTTPS |
| Auth | Mobile session token via bridge | Auth token from `LaunchContextV1.authToken` (provisioned by portal at lab launch) |
| Offline buffering | Mobile RN app's existing queue | Embedded queue (Capabilities.Defaults.Telemetry.Core) |
| Lives in | `Capabilities.Defaults` | `Capabilities.VR.Telemetry` (sub-package, mirrors `Capabilities.VR.Fusion` pattern from §9.8) |

**Shared offline-queue core.** The retry/batching/flush behavior is identical across implementations. The mobile `telemetry-queue.service.ts` is the canonical reference; `Capabilities.Defaults.Telemetry.Core/` ports its semantics into Unity (C#):

- Exponential backoff with jitter on transport failure
- Batching window (default 5s) with size-threshold flush (default 50 events) and forced-flush triggers (app foreground/background, lab end, explicit `Flush()` call)
- Idempotency keys per event (UUID v7 with embedded timestamp) so retried events deduplicate at the portal
- Bounded local buffer (default 10,000 events / 5 MB, whichever first) with oldest-first eviction; eviction emits a synthetic `TelemetryBufferEvictedEvent` so analytics can detect data loss
- Anon Supabase key + per-session auth token (refreshed on token expiry, with offline buffering across token refresh failure)

**Existing VR scripts.** The `HealthOn VR/shell/` folder in the consumer repo contains in-progress telemetry scripts. These are pulled into `Capabilities.VR.Telemetry` as the canonical `DirectCloudQueuedTelemetryService` implementation, with anything not yet covered ported from mobile's queue logic. The interface contract and envelope are non-negotiable — implementation choices below the interface are flexible.

**Result.** The same `TelemetryEventV1` payload reaches Supabase from any surface (AR via mobile, VR direct, future web direct). Portal ingestion is one path; analytics queries do not branch by surface. This is the substrate for the "all surfaces, one data plane" outcome promised in the broader product strategy.

**Future surfaces.** Web shell, alternate VR headsets (Pico, Vision Pro), or any other host implements the same `ITelemetryService` interface against the same `Telemetry.Core` queue module. Adding a surface adds a transport implementation; analytics-side code never changes.

**UI Toolkit conventions (adopted from Unity 6 manual — UI Toolkit has materially improved since 2022.3):**

- **Separation of concerns** — UXML defines structure; USS defines style; C# defines behavior and data binding. No inline styles when a USS file can hold them (memory efficiency — inline styles are per-element).
- **BEM class naming for custom controls** — every DevKit control carries its type class plus a modifier when applicable (e.g., `devkit-button`, `devkit-button--primary`, `devkit-section__header`). Matches Unity's own convention (`unity-button`, `unity-text-element`).
- **Case discipline** — UXML attribute names are `kebab-case`; C# property names are `PascalCase` (or `camelCase` for private fields). A C# property `MyFloat` appears in UXML as `my-float`. Enforced by a Roslyn analyzer rule at CI.
- **Custom controls use `[UxmlElement]` + `[UxmlAttribute]` (Unity 6), NOT the legacy `UxmlFactory` + `UxmlTraits` pattern.** The new attributes are source-generator-backed, reduce boilerplate, and are the canonical pattern going forward. Spec invariant: no `UxmlFactory<T>` class in any DevKit editor code.
- **Unity 6 Runtime Data Binding system** — bind C# properties directly to UI elements in UI Builder, no manual "glue code." Used for every property-panel surface (LabConsoleEditor, StepInspectors, Building Block metadata editor). Hand-rolled `onChange` wiring is forbidden in new windows except for controls that fundamentally don't fit the binding model (drag-drop, custom drawing).
- **Hub uses Unity 6's `TabView` + `Tab` controls** (introduced in UI Toolkit 1.0 / Unity 6) for the tabs enumerated in §13.2. No custom tab widget.
- **Icon support on Button, ListView, TreeView** (Unity 6 addition) used for Hub actions and building-block thumbnails.
- **UI Debugger use** — every new window is validated under `Window → UI Toolkit → Debugger` for hierarchy cleanliness, unused styles, and redundant class assignments before PR. A check-list item on the Authoring PR template.

## 15. The non-breaking evolution contract

### 15.1 The threat model

Unity serializes content into scenes, prefabs, and ScriptableObject assets using a three-part type identity: `Assembly + Namespace + ClassName`. Naive refactoring (rename, namespace move, asmdef move) breaks serialized data. Shipped AR/VR labs would become null-step scenarios at runtime — silent breaks shipped to universities.

### 15.2 The five rules

1. **Type identity is frozen.** Relocating a type that ships in a prefab/scene/asset requires `[MovedFrom]` coverage (`using UnityEngine.Scripting.APIUpdating`). `[MovedFrom]` support for `[SerializeReference]` identity landed in Unity 2022.1 and carries forward into Unity 6. Verified by fixture load test (Gate 2). **Note:** v0.10 fixtures were captured in Unity 2022.3; they are loaded and verified in Unity 6 as part of the P1 fixture-capture pass, which proves the Unity 2022.3 → Unity 6 migration does not itself break serialized content.

   **Known Unity caveat — prefab overrides.** Per Unity's own issue tracker, `[MovedFrom]` is **not respected** when a `[SerializeReference]` object is created inside a prefab *override* that lives in a PropertyModification; data can be lost in that narrow scenario. Mitigation shipped in P1:
   - A pre-relocation validator (`PrefabOverrideSerializeReferenceValidator`) scans every prefab for SerializeReference-typed override entries and emits a report. Any detected override is surfaced in the Hub with a one-click "materialize into prefab root" fix that writes the override into the parent prefab, preserving data before the type move.
   - The migration test corpus includes at least one prefab with a SerializeReference override to exercise this path in CI; fixtures capture **before** and **after** states.
   - If the validator reports any detected override that cannot be safely materialized, the PR moving the type is blocked until the affected prefab is resolved. This is stricter than Unity's default behavior and is deliberate — silent data loss in a shipped university lab is unacceptable.
2. **Field identity is frozen.** Renames require `[FormerlySerializedAs]`. Type changes not allowed without a `ScriptedImporter` migrator. Field additions require non-destructive defaults.
3. **MonoBehaviour public contract is frozen.** Public methods + `SerializeField` fields on types attached to shipped prefabs form the public API surface. CI gate compares against `publicApi-v0.10.baseline.txt` and fails on removal/rename.
4. **Schema contracts are frozen per version.** All 15 `V1` contracts are immutable after publish. `V2` ships alongside with a migrator. Mismatched version errors loudly.
5. **Runtime behavior is frozen for equivalent inputs.** Step runners produce identical `StepExited` outcomes for identical step data across minor versions. Verified by golden replay tests.

### 15.3 The seven CI gates

Three non-breaking-contract gates:

- **Gate 1 — Public API surface diff:** parses `publicApi-v<version>.baseline.txt`, reflects over `Pitech.XR.*` public surface, asserts no removal/rename
- **Gate 2 — v0.10 fixture load:** opens curated v0.10 scenes from `Tests/Fixtures/v0.10/`, walks `Scenario.steps`, asserts no null entries and no missing script warnings; also runs `PrefabOverrideSerializeReferenceValidator` on every fixture
- **Gate 3 — Golden replay:** replays scripted inputs against v0.10-built scenarios, compares event stream to `Tests/Fixtures/golden/<scenario>.events.json`, fails on divergence

Three architectural gates:

- **Gate 4 — Zero-alloc assertion:** hot paths measured allocation-free via `Assert.AllocFree` custom constraint
- **Gate 5 — Dependency direction:** asmdef graph parsed, no upward reference across layers
- **Gate 6 — Roslyn banned APIs:** runtime assemblies scanned for `Camera.main`, `GameObject.Find`, etc.

One IL2CPP-safety gate:

- **Gate 7 — IL2CPP round-trip:** every V1 schema contract (LaunchContextV1, LabObservationV1, LabActionV1, ScenarioV1, LabConsoleV1, TelemetryEventV1, AgentToolCatalogV1, ConsentRecordV1, BuildingBlockMetadataV1, AttemptIdentityV1, LabObjectiveV1, StepObjectiveV1, ObjectiveOutcomeV1, ObjectiveEvaluationProposalV1, AttemptSummaryV1) is serialized and deserialized inside an IL2CPP-built player target; asserts round-trip equivalence with the editor-Mono round-trip. Catches Newtonsoft.Json AOT stripping issues before they hit device builds.

**All 7 gates green = merge allowed.** Otherwise no merge.

### 15.4 Last-resort break policy

If a change truly cannot be made without breaking shipped scenarios (planned: v2.0.0 or later):

1. Announce in CHANGELOG one minor version ahead with `[Obsolete]` warnings
2. Ship a menu-driven migration tool with dry-run
3. Re-capture fixtures as `Tests/Fixtures/v<major>/` after migration
4. Major version bump
5. CHANGELOG flags the break with migration link

### 15.5 No hardcoding — the commitment per subsystem

| Don't ship | Ship instead |
|---|---|
| `else if (step is QuizStep)` branch chains | `StepRunnerRegistry` + `IStepRunner<TStep>` |
| Reflection-by-string (`GetField(memberName)`) | Typed `IConditionValueSource`, typed `IEffectHandler<TEffect>`, typed `Parameter<T>` |
| `#if ENABLE_INPUT_SYSTEM` sprinkled in runtime | `IInputService` capability; ifdef lives only in capability impl |
| `PhotonNetwork.InRoom ? photonView.RPC(...) : Apply(...)` | `INetworkReplicationService` capability; Fusion impl in optional sub-package |

| Observation pulse overhead | < 0.5 ms/frame | PlayMode test |
| Fusion tick rate | 30 Hz default | Configured in `FusionReplicationService` |

**Performance regressions block merge** — just like behavioral regressions. The budget is a contract, not a goal.

## 18. Package versioning and release discipline

### 18.1 SemVer commitment

From v0.11 onward, DevKit follows SemVer 2.0, voluntarily stricter than the spec requires for 0.x.y:

- **`MAJOR.minor.patch`** — breaks Rules 1–5. Only planned major version in this trajectory: `0.x → 1.0`.
- **`major.MINOR.patch`** — additive only. Applies to every phase bump v0.11 → v0.16.
- **`major.minor.PATCH`** — bug fixes, doc, tests; no API surface change.

**At v1.0.0** the public API surface freezes. `publicApi-v0.10.baseline.txt` becomes `publicApi-v1.0.baseline.txt`; CI gates against removal/rename forever after.

### 18.2 Git branching strategy

- `main` — active development, tracks whichever phase is in flight
- `legacy-0.10.x` — last v0.10 commit + any critical-fix patches backported; maintained indefinitely
- `release/v0.11`, `release/v0.12`, … — cut from `main` at each phase exit, tagged, merged back
- Feature branches — conventional `feature/<short-description>` per PR

### 18.3 Version switching from Unity

**Near-term (git tags):**

AR / VR repos reference an explicit tag in `Packages/manifest.json`:

```json
"com.pitech.xr.devkit": "https://github.com/Smalers1/pitech-xr-devkit.git#v1.0.0"
```

Switching versions = editing one line. Reverting = editing one line back. Unity Package Manager re-resolves on save (~10 s).

**Post-v1.0 (scoped registry, optional):**

Publish to a scoped registry (self-hosted Verdaccio or OpenUPM). `manifest.json` adds a `scopedRegistries` entry; Package Manager UI shows a version dropdown with "Update to" button; semver ranges become available (`"1.x"`). Git repo remains source of truth; CI pushes to registry on every tag.

### 18.4 Deprecation policy

Any public API flagged `[Obsolete]` lives one full minor version before removal. CHANGELOG entry on deprecation. Consumer receives compile warning during the deprecated window.

### 18.5 Rollback

Consumers pin a prior tag at any time. Patches are backportable to the prior minor for at least 2 minors (e.g., v0.14 ships → v0.12.x still receives critical fixes until v0.15 lands).

## 19. Testing strategy

### 19.1 Test stratification

| Tier | Scope | Speed | Cost |
|---|---|---|---|
| **EditMode unit (NUnit)** | Domain, Runtime (with fake capabilities), Core (bus, registries) | ms | cheapest — tens per PR |
| **EditMode integration** | ValidatorRegistry, Authoring pipelines, AgentToolCatalogBuilder | < 1s | cheap |
| **PlayMode integration** | ScenarioRunner + real capabilities, LabConsoleRuntime + Fusion | seconds | moderate |
| **PlayMode migration** | v0.10 fixture load, `[MovedFrom]` verification, facade equivalence | seconds | moderate |
| **PlayMode golden replay** | Scripted input replay vs recorded event log | seconds | moderate |
| **PlayMode performance** | Oculus Perf HUD integration, allocation budgets | seconds | moderate |
| **MPPM multiplayer** | Fusion 2 Shared-Mode state-authority routing, 2-to-4 virtual peers, in-editor | seconds | moderate |
| **IL2CPP round-trip** | Every V1 schema contract, built to IL2CPP player, serialize/deserialize parity | seconds | moderate (Gate 7) |
| **On-device smoke** | Build to Quest / Android; play a scenario end-to-end | minutes | expensive — per-phase, not per-PR |

**Test Framework 2.0 (Unity 6) conventions adopted verbatim:**
- Prefer `[Test]` over `[UnityTest]` unless a frame yield or timing wait is actually needed. Avoids coroutine overhead.
- Use `[RequiresPlayMode]` to co-locate EditMode and PlayMode tests in the same assembly where the subject warrants it (e.g., a runner's logic is EditMode-testable but its lifecycle requires PlayMode — one assembly, two attributes).
- Use `async Task` tests for async-returning subjects — supported in Test Framework 2.0, cleaner than coroutine `[UnityTest]`.
- Keep test assemblies separate per layer (`Domain.Tests`, `Core.Tests`, etc.) so consumers don't accidentally pull test dependencies into production builds.

**Multiplayer Play Mode (MPPM) — local multiplayer testing in Unity 6:**
- Replaces ParrelSync for DevKit's Fusion testing needs.
- Spawns up to 4 virtual player instances from the editor without needing multiple editor windows or separate machines.
- Integrates with Fusion 2's Shared Mode: one virtual instance is the StateAuthority host; the others are non-authority peers. Tests the trust-but-verify RPC routing (§9.8) without a Quest build.
- Deliverable in P3: MPPM-based PlayMode integration test that validates: non-authority peer invokes `offer_hint_current_step` → RPC to authority → authority re-runs gates → `[Networked]` property updates → both peers observe via `ChangeDetector` → audit records origin + authority verification.

### 19.2 Fixture strategy

- `Tests/Fixtures/v0.10/` — 5–8 curated v0.10 scenarios (cue-cards-heavy, quiz-heavy, selection-heavy, group-step, insert step, conditions-routed, addressables-spawned, timeline-synced). Captured at start of P1. Never modified.
- `Tests/Fixtures/golden/` — event streams captured from v0.10 behavior. Recorded at start of P2. Golden replay compares against these.
- `Tests/Fixtures/sample-labs/` — small labs used by Scenario Simulator and PlayMode integration. Maintained as the architecture evolves.

### 19.3 Performance testing

- Oculus Performance HUD values captured as PlayMode test output
- Allocation budgets asserted via `Assert.AllocFree` custom NUnit constraint
- Per-frame ms budgets asserted per capability path
- Regressions fail the PR (Gate 4)

### 19.4 What we do NOT test

- Unity engine behavior itself (we trust Unity's test coverage)
- External networking (stub Photon Fusion in PlayMode tests; real Fusion only in on-device smoke)
- Vuforia / ARFoundation tracking (capability impls tested via fakes; real tracking in on-device smoke only)
- LLM behavior (VICKY's reasoning is tested in the web portal repo; DevKit tests the substrate that receives VICKY's tool calls)

## 20. Deferred / explicitly out of scope for v1.0

The following are intentionally not in v1.0 scope. Each has a clear path for later inclusion.

- **Full ARFoundation impl.** The capability seam (`ICameraService`, `IWorldAnchorService`, `ISelectablesRuntime`) makes this a P8+ drop-in.
- **Web Lab Player consumer.** Capability seam supports it; Unity WebGL target requires additional capability impls; no authoring work required.
- **Localization beyond a no-op default.** `ILocalizationService` exists; real impl in 1.1+ without API change.
- **Full scoped registry publishing.** Git tags carry through v1.0; registry post-v1.0 is pure polish.
- **`VickyMode.LabAuthor` and prompt-to-simulation.** 2028 scope. Substrate is ready; role + actions + metadata index added then, no architecture change.
- **`IVoiceService` and `IVisualAgentPresence` capabilities.** 2027 scope. Capabilities added as new interfaces, no architecture change.
- **Content-safety gate real impl.** Slot reserved; default is no-op in v1.0. Activated when agent-authored content first needs safety review (anticipated 2028).
- **PUN → Fusion migrator for existing VR labs.** Per-lab migration tracked in handbook §6.7 "Fusion Migration" epic. DevKit v1.0 supports Fusion natively but does not auto-migrate PUN scenes.
- **iOS UaaL.** Mobile app ships Android-only today; iOS is a 2026 H2 item per handbook §9.3.
- **Full JSON authoring of Scenarios from web portal.** `ScenarioV1` JSON contract ships in v1.0 so the day a web editor is built, the shape already exists. Editor itself is post-v1.0.
- **Source generators for registry wiring.** Unity 6 supports `IIncrementalGenerator`-style Roslyn source generators (label the DLL `RoslynAnalyzer` in Unity). We could generate typed accessors for every registered `[StepRunner]`, `[EffectHandler]`, `[ConsoleAction]` class — eliminating reflection at startup. v1.0 uses reflection-once-at-domain-reload because the current scale (< 50 registered classes) doesn't justify the source-generator complexity. Reserve as a post-v1.0 optimization if startup time or cold-start allocation becomes material.
- **Third-party DI container (VContainer, Reflex, Zenject).** Our hand-rolled `CapabilityRegistry` is intentional. Swapping to Reflex (fastest, IL2CPP-safe, source-generator-backed) or VContainer (widely adopted, stable) would save ~200 lines of registry code at the cost of a mandatory consumer dependency and a new API surface for the AR/VR teams to learn. Reserve. If we ever need constructor injection across 100+ services, revisit.
- **Custom Awaitable scheduler.** Unity 6's `Awaitable` is sufficient for everything in v1.0. If we ever need PlayerLoopTiming-style fine-grained control, UniTask's internals are the pattern to study.

## 25. Appendix C — Glossary

| Term | Definition |
|---|---|
| **Lab** | An Addressables-delivered prefab containing a `LabRoot` plus optionally a `Scenario`, a `LabConsole`, and interactables. |
| **LabRoot** | The entry-point MonoBehaviour on a lab prefab. Creates lab-scoped event bus and capabilities; starts runtime. |
| **Scenario** | Linear step graph. One of the three lab behaviors. |
| **LabConsole** | Dynamic parameter store + declared actions. One of the three lab behaviors. Replaces VR's ControlOptionManager pattern. |
| **ConsoleParameter** | A typed, live-controllable value (e.g., vitals heart-rate). |
| **ConsoleAction** | A declared operation with role tags. Auto-exposed to professor panel and VICKY tool catalog. |

| **Capability** | A typed runtime service the host provides to the lab. 18 interfaces in v1.0. |
| **Objective** | An author-declared definition of what counts as success in a lab (or step / console). Polymorphic criterion family across three tiers (deterministic / state-goal / AI-evaluated). Engineer-facing term; portal presents these as "Competence" or "Learning Outcomes" for buyers. See §7.7. |
| **Constraint** | An author-declared anti-objective — something that must NOT happen (sterile-field violation, missed safety check). Polymorphic family in `Domain/Objectives/Constraints/`. Violations publish `ConstraintViolatedEvent`. |
| **Objective tier** | `Tier1_Deterministic` (author specifies full criteria, runtime evaluates) / `Tier2_StateGoal` (author specifies goal states + constraints, runtime evaluates) / `Tier3_AiEvaluated` (author specifies prompt + category, VICKY in Examiner mode proposes, professor confirms). |
| **LabRoot.Capabilities** | The lab's view over capabilities: lab-scope first, app-scope fallback. |
| **DevKitBootstrapper** | Per-consumer (AR, VR, Simulator) base class that registers capability impls once per app. |
| **LabEventBus** | The in-process pub/sub bus, one per lab, tenant-scoped by construction. |
| **StepRunner** | One per step type. Runs a step. Emits events. Projects observations. |
| **EffectHandler** | One per effect kind. Applies a parameter change's effect (timeline speed, animator param, etc.). |
| **Observation** | A versioned, typed snapshot of lab state that VICKY reads. |
| **Gate pipeline** | Consent → Role → Rate → ContentSafety → Audit. VICKY actions flow through it. |
| **Facade (SceneManager)** | The v0.10 `Pitech.XR.Scenario.SceneManager` kept at its current namespace + class name, internally delegating to v1.0 runtime. |
| **Fixture** | A v0.10 scene or recorded event stream used in CI gates. Never mutated. |
| **BuildingBlock** | A reusable prefab with a `BuildingBlockMetadataV1` sidecar. Currency of the 2028 prompt-to-simulation vision. |

## 28. Addendum — Domain & content systems (board, 2026-06-09)

> **Status: board-added 2026-06-09, additive, pending Heisenberg/Stergios review.** This addendum does not alter
> any §1-27 decision. It *confirms and extends* seeds already in this spec with four decisions taken in the
> 2026-06-09 launch-planning session (see `Documentation~/plans/2026-06-09-devkit-launch-plan.md` and the
> `_after-launch/` plan). Each is grounded in a read of the existing HealthOn VR systems it builds on.

### 28.1 Ownership — multiplayer + localization logic live IN DevKit

**Decision (Petros, 2026-06-09):** the multiplayer state logic, the localization pipeline, **and their editor
automations** live inside `com.pitech.xr.devkit` (consumed by both HealthOn AR and HealthOn VR). Only **VR-Shell-
specific** scripts (the shell menu, `EntitlementsClient`, the cloud-catalog card UI) stay outside DevKit in HealthOn
VR. Rationale: multi-platform reuse — AR has no localization today and no multiplayer authoring; both belong to the
shared toolkit, not one consumer app. This is consistent with the layer model (§6): these are Capabilities/Domain
concerns, and the consumer apps are thin.

### 28.2 Multiplayer — `NetworkedStates` is the existing flow-store backend

Confirms §9.8 (Fusion 2 Shared-Mode StateAuthority) + `INetworkReplicationService` (Appendix A #11). The existing
HealthOn VR `Assets/Scripts/Multiplayer/NetworkedStates` is a **mature, idiomatic Fusion 2 Shared-Mode fact store** —
a `[Networked] NetworkDictionary<NetworkString<_64>, NetworkBool>` (`NetworkStateManager`) of named boolean facts,
an object-scoped `NetworkedInteractionRelay` (latch + monotonic pulse, `ChangeDetector`-edged so one-shots don't
replay on resimulation), and plain-MonoBehaviour listeners/triggers that poll replicated facts. It is **already the
"sync facts, not the step pointer" model**: a late joiner rebuilds from the replicated dictionary on `Spawned()`.

- **It moves into a DevKit `Networking` module** (the WS-6 reserved slot, Fusion-gated) and is wrapped behind the
  planned **`IScenarioFlowStore`**: `IsStepComplete(id) => GetState(id)`, `CompleteStep(id) => SetStateTrue(id)`,
  `StepFactChanged` off the existing edge-detection. The "Make Multiplayer" **editor automation** (rig + Fusion-object
  injection at author-time) moves in too.
- **Two gaps to design (post-launch):** (1) facts are boolean-only — richer outcomes (correct/wrong/score/branch)
  need the value type widened or encoded as multiple keys; (2) the runner's step pointer is **not yet fact-driven** —
  each peer's `SceneManager` runs `Run()` from idx 0 locally, so a late joiner's *runner* does not fast-forward. That
  is exactly the §17 arch-P2 work (the extracted runner consults `IScenarioFlowStore` and skips completed steps).
- `LabConsoleRuntime` (§8.6) sits **above** this fact store; on the authority it translates a professor parameter
  change into the replicated store. Converge the vitals `ControlOptionManager` (currently PUN) onto this Fusion path.

### 28.3 Localization — extend `ILocalizationService` into a cloud content pipeline

Extends `ILocalizationService` (Appendix A #15, today "i18n of UI copy") into a full **content layer**: lab **text +
audio narration**, keyed by `(contentKey, language) -> {text, audioUrl}`, **Web-Portal-managed, VICKY-translated,
ElevenLabs-audio-generated, runtime-fetched-by-language**. This is the EU-scale moat ("Germany hospital, 4 labs,
translated in minutes").

- **Build on the existing VR-Shell cloud-fetch pattern** (`EntitlementsClient` + `LabCardUI.LoadCoverCoroutine`):
  UnityWebRequest + Bearer JWT + an edge-fn projector + Supabase Storage public URLs + a `LabMetadataCache`
  version-cache. The cover-texture coroutine maps almost line-for-line to audio (`UnityWebRequestMultimedia.GetAudioClip`).
- **Reuse the existing localization key model + translate prompt.** The HealthOn VR localization pipeline
  (`Assets/Scripts/Editor/Localization/`) already keys strings by stable `GlobalObjectId` and ships a
  **medical-terminology, placeholder-preserving LLM-translate prompt** (`ManualTranslationIO`). The keys back a cloud
  `(key, lang)` table directly; the prompt is reused **verbatim** by the VICKY auto-translate edge function. This
  pipeline **moves into DevKit** (§28.1).
- **Trap:** do not keep build-baked StringTables AND cloud as two sources of truth. The cloud resolver becomes
  canonical; baked tables become the offline fallback / launch bootstrap.
- **Launch vs post-launch:** at launch, ship **keyed Greek + English** via the (relocated-into-DevKit) build-baked
  pipeline — and AR gets localization for the first time. The cloud self-serve pipeline (VICKY translate + human fix
  + ElevenLabs + runtime fetch-by-language + a Web-Portal lab-content editor) is **post-launch**. The launch
  requirement is only that lab/analytics text is **keyed** now, so the cloud resolver slots under the same keys later.

### 28.4 Vitals — the patient digital twin

Formalizes the glossary's `ConsoleParameter` example ("a typed, live-controllable value, e.g. vitals heart-rate")
and `LabConsole`'s "replaces VR's ControlOptionManager pattern" into a typed **Vitals** domain system. Today the
physiology logic is scattered across ~8 HealthOn VR scripts (breathing blendshape driven by a Timeline whose *speed*
is the vital value; pulse audio gated by the stethoscope; skin-colour controller; etc.) with an **untyped**
`ControlOptionManager` dispatcher (on PUN, not Fusion).

- **A `Vital` = a typed `ConsoleParameter` + a 3D BINDING** (blendshape / timeline-speed / material / audio-pitch
  driver). A `PatientVitals` component is the single typed model (pulse / breathing / BP / temp / wounds / skinColour…).
- **It implements `IAgentStateSource`** (the `ILabObserver` feed, §10.5 / Appendix A #16) — so VICKY reads structured
  patient state for free. The empty `IAgentStateSource` seam the P5 substrate-observer slice left is the natural home.
- **Cascade rules** (low pulse -> pale skin) as derived bindings; **ScriptableObject profiles** author a patient in
  minutes (= a reusable digital twin). Vitals replicate as LabConsole parameters (§9.8).
- **Launch vs post-launch:** at launch, lay the **typed foundation** (the `PatientVitals` component + the binding
  model + the `IAgentStateSource` implementation), additively, alongside the existing scattered logic. The full
  digital-twin (cascade rules, profile library, migration of the scattered scenes, ControlOptionManager convergence)
  is **post-launch**.

### 28.5 AI-assisted authoring — the content-acceleration principle

The unifying reason §28.2-28.4 (and the analytics/Action model) all use the same shape — *typed data, referenced by
a stable key, resolved externally* — is that it makes content **AI-authorable**. Every authoring artifact
(`ScenarioV1`, the analytics/Action config, a `PatientVitals` profile, a translation set) is **clean schema'd JSON +
a deterministic editor "apply" command + a headless entry**. Claude/AI drafts the data; a developer applies it via a
DevKit Hub command and reviews it in the visual tool (Scenario Graph, the Binding Sheet, the Vitals inspector).

- The DevKit Hub (§13) provides a **library of "apply this AI-authored artifact" commands** (Import Scenario, Load
  Patient Profile, Import Translation Set, Export-Lab-as-Fixture). Each command has a headless entry so an agent or
  CI can apply without a human clicking.
- VICKY is the **translation engine** (§28.3); Claude is the **scenario / vitals-profile / analytics drafter**. The
  eventual CLI/MCP bridge lets an agent drive generate -> apply -> `Evaluate Changes` -> report end-to-end.
- This is post-v1.0 as a *system*, but the **seam is the JSON-first + keyed-content + typed-parameter architecture
  this spec already mandates** — so no new architecture is required to reach it, only the apply-command library.

- **No package-version bump in launch Phase A** (Stergios): `package.json` `version` stays `0.10.5` through the

- **The package-metadata cutover is ONE explicit Phase D (post-launch) step:** the `unity` field 2022.3 -> 6000.0 AND the
